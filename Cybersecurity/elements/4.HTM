A while ago, I curled up in bed, with my laptop and a coffee, and scraped every single IDOR report from hackerone. A week or so later, I had compiled notes and findings from my reading and I'm here to show you what I've learnt, and how my understanding of a seemingly simple bug has changed massivley.

前些日子，我蜷缩在床上，带着笔记本电脑和咖啡，从 HackerOne 爬取了每一份 IDOR 漏洞报告。大约一周后，我整理好了阅读笔记和发现成果，现在我要向你们展示我的学习收获，以及我对这个看似简单的漏洞的认知发生了怎样巨大的转变。

For reference, throughout the article I will refer to various bug reports via numbers, these numbers correlate to bug reports, which can be found by replacing XXXXXX in the url: https://hackerone.com/reports/XXXXXX

作为参考，本文提及的漏洞报告编号对应具体报告，可通过替换下方 URL 中的 XXXXXX 查看：https://hackerone.com/reports/XXXXXX

Your understanding of IDOR is wrong.

你对 IDOR 漏洞的理解是错误的。

That subheading was pretty dramatic, realisitcally, it should be "Your understanding of IDOR is too narrow". Typically people think an IDOR is just finding a parameter called "ID" (or similar) and replacing the number with another, like report 797685.

那个小标题确实很戏剧化，但现实点说，它应该改为"你对 IDOR 的理解过于狭隘"。通常人们认为 IDOR 漏洞就是找到一个名为"ID"(或类似)的参数并替换其中的数字，比如报告 797685 所述案例。

Then receiving Status 200, saying an operation from a different user has been complete.

随后收到 Status 200，提示另一用户的操作已完成。

IDOR is wider than this. IDOR does not have to apply to accounts. It doesnt not have to apply to IDs. It is not just found in basic formats. We will discuss this misunderstanding in a bit more detail.

IDOR 的范畴远不止于此。它并不局限于账户场景，也无需与 ID 参数强关联，更不会仅存在于基础格式中。我们稍后将对此误解展开更详细的探讨。

Where do are IDORs commonly found?

IDOR 漏洞通常潜伏于何处？

When reading these reports, I counted where I found the various weakness emerged and here are the results:

通过研读这些漏洞报告，我对各类弱点的产生场景进行了系统统计，以下是分析结果：

  1. REST APIs 31.8%  REST 接口 31.8%
  2. GET parameters 25.8%  GET 参数 25.8%
  3. POST request bodies 21.2% POST 请求体中的 ID 占 21.2%
  4. graphQL endpoints 9.1%  GraphQL 端点 9.1%
  5. PUT parameters 4.5%  PUT 参数 4.5%
  6. IDs in the request header 3.0% 请求头中的 ID 占 3.0%
  7. IDs in the cookies 3.0% Cookie 中的 ID 占 3.0%
  8. Misc Query langauges 1.5% 其他查询语言中的 ID 1.5%

"IDOR is a low hanging fruit which takes no skill"

“IDOR 是一个无需任何技术含量的低垂果实”

This is often an idea which I hear propagated; it is a fairly reductionist way of thinking about this bug. Although it may be more technically simple than most other bugs, there are still concepts you can learn to increase your success with them. And it is still the bug which yeilds higher bounties, than Privesc, SQLi and Code Injection.

这是我常听到的一种被广泛传播的观点，这种对漏洞的理解方式带有明显的简化主义色彩。尽管 IDOR 漏洞的技术复杂度可能低于大多数其他漏洞类型，但其中仍有许多可学习的技巧能助你提升发现成功率。更何况，相较于权限提升、SQL 注入和代码执行等漏洞，它仍是漏洞赏金回报率更高的漏洞类型。

Here are some points to keep in mind:

以下是一些需要牢记的要点：

  • Can you find IDs that arent incremental? Sometimes an ID can be found by visiting a page or an asset.

    你能找到非递增的 ID 吗？有时候，访问某个页面或资源就能发现 ID。

An ID can be found in the source code on a page in a twitter:

在 Twitter 的某个页面的源代码中可能会发现 ID。

  • Is the ID encoded or not in plaintext? (291721)

    ID 是以明文形式编码的吗？(291721)

  • Can a file/resource be accessed directly from the url, without needing prior authentication? (94790, 258260, 230328, 230870, 126861)

    能否无需事先身份验证，直接通过 URL 访问文件/资源？(94790、258260、230328、230870、126861)

Images are commonly vulnerable to this kind of vulnerability, as seen in report 258260:

图片通常容易受到此类漏洞的影响，如报告 258260 所示：

  • Are there bypasses, like finding similar sites with less protection (876300, 715054, 271393)

    是否存在权限绕过机制，例如发现防护较弱的类似站点（如 876300、715054、271393 号案例）？

Some alternate sites of the main one may contain vulnerabilities which were patched in the main one. Try different regional top level domains, or APIs on different

主站的某些替代站点可能包含主站已修复的漏洞。可尝试不同地区的顶级域名或不同子域上的 API 接口进行测试。

  • Can you get an IDOR by changing the request method, or does the exploit involve using a different method? (204984, 199321, 297751)

    能否通过更改请求方法（如 204984、199321、297751 号案例）实现 IDOR 漏洞利用，或是此类漏洞利用需涉及不同方法的调用？

  • Are IDs not actually numbers but strings? Sometimes the IDOR can be exploited via a username or email change instead of an ID. (262661, 152407, 587687)

    ID 并非数字而是字符串？有时通过用户名或邮箱修改而非 ID 修改即可利用 IDOR 漏洞。(262661, 152407, 587687)

What is the impact of IDOR? Adjust your expectation.

IDOR 漏洞的影响是什么？调整你的预期。

IDOR is probably the easiest way to get a high priority account takeover. However, tunnel vision-ing this objective can stop you from spotting IDORs which results in different, or lower priority IDORs. Instead, think about:

IDOR 或许是实现高优先级账户接管的最简单途径。然而，若只盯着这一目标，反而可能让你错过那些会导致不同类型或优先级较低的 IDOR 漏洞。不妨考虑以下方面：

  • Does it leak PII (293490, 980511, 723461, 668439, 783708, 439729, 152407)

    是否涉及 PII 泄露（案例号：293490、980511、723461、668439、783708、439729、152407）

293490 Shows us how an IDOR can result in leaking emails of all users in a response. Leaking PII.

案例 293490 向我们展示了 IDOR 漏洞如何导致响应中泄露所有用户的电子邮件，进而造成个人身份信息(PII)的泄露。

  • Can you bypass payment (391092)

    你能绕过支付验证吗（391092）

  • Can you do actions on other's behalfs (1005020, 725569, 258260)

    能否代表他人执行操作（1005020, 725569, 258260）

  • Can you destroy or damage any assets or info? (156537, 264754, 153905, 120115)

    能否销毁或损坏任何资产或信息？（156537, 264754, 153905, 120115）

Summary/Key takeaways  总结/关键要点

  • IDOR bugs are broader than most people think.

    IDOR 漏洞的涵盖范围比大多数人想象的要广泛。

  • Almost 80% of IDORs are found in REST APIs, GET parameters or POST request bodies, although you should still search in the other 20%.

    虽然仍有 20%的可能性存在于其他位置，但近 80%的 IDOR 漏洞出现在 REST API、GET 参数或 POST 请求主体中。

  • Success in finding IDORs comes from creative ways of finding them.

    成功发现 IDOR 漏洞的关键在于寻找它们的创造性方法。

  • Your goal is not always to get P1 Account takeovers.

    你的目标并非总是要达成 P1 级别的账户接管。
